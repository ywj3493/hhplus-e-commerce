## 개요

### 문서 목적

본 문서는 E-Commerce 백오피스(Admin) 기능의 요구사항을 정의합니다. Admin 기능은 사용자 대면 기능과 비즈니스 도메인이 다르므로 별도 문서로 관리합니다.

### Admin 시스템 특성

- **목적**: 상품, 재고, 쿠폰, 주문 등의 백오피스 관리
- **사용자**: 관리자 (Admin)
- **인증**: Admin 전용 인증/인가 시스템 필요
- **아키텍처**: 현재는 동일 서비스 내 구현, 향후 마이크로서비스 분리 가능성 고려

### 범위

**포함 (In Scope)**:
- 상품 관리 (등록/수정/삭제/활성화)
- 상품 카테고리 관리
- 상품 옵션 관리
- 재고 관리 (입고/조정/이력)
- 쿠폰 발급 관리 (생성/설정/현황)
- 주문 관리 (조회/상태 변경/취소)

**제외 (Out of Scope)**:
- 상품 이미지 업로드 및 관리 (추후 구현)
- 고급 통계 및 대시보드
- 데이터 전송 모니터링
- 관리자 계정 관리 (이미 구현되어 있다고 가정)

## Admin 기능 요구사항

### FR-ADMIN-PROD: 상품 관리

| ID | 요구사항 | 우선순위 | 상태 |
|----|---------|---------|------|
| **FR-ADMIN-PROD-01** | 시스템은 상품을 등록할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-PROD-02** | 시스템은 상품 정보를 수정할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-PROD-03** | 시스템은 상품을 삭제할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-PROD-04** | 시스템은 상품을 활성화/비활성화할 수 있어야 함 | MUST | ⏳ 대기 |

#### 상세 설명

**FR-ADMIN-PROD-01: 상품 등록**
- **입력**:
  - 상품명
  - 가격
  - 설명
  - 카테고리 ID
- **처리 과정**:
  1. 입력 데이터 유효성 검증
  2. 카테고리 존재 여부 확인
  3. 상품 정보 저장
  4. 초기 재고 0으로 설정
  5. 초기 상태: 비활성화 (INACTIVE)
- **출력**: 생성된 상품 ID 및 상품 정보
- **관련 유스케이스**: [UC-ADMIN-PROD-01](user-stories.md#uc-admin-prod-01-상품-등록)

**FR-ADMIN-PROD-02: 상품 수정**
- **입력**:
  - 상품 ID
  - 수정할 필드 (상품명, 가격, 설명, 카테고리 ID)
- **처리 과정**:
  1. 상품 존재 여부 확인
  2. 입력 데이터 유효성 검증
  3. 카테고리 변경 시 카테고리 존재 여부 확인
  4. 상품 정보 업데이트
- **출력**: 수정된 상품 정보
- **관련 유스케이스**: [UC-ADMIN-PROD-02](user-stories.md#uc-admin-prod-02-상품-수정)

**FR-ADMIN-PROD-03: 상품 삭제**
- **입력**: 상품 ID
- **처리 과정**:
  1. 상품 존재 여부 확인
  2. 주문 내역 확인 (소프트 삭제 고려)
  3. 상품 삭제 또는 비활성화
- **출력**: 삭제 확인 메시지
- **제약사항**: 주문 내역이 있는 상품은 소프트 삭제 권장
- **관련 유스케이스**: [UC-ADMIN-PROD-03](user-stories.md#uc-admin-prod-03-상품-삭제)

**FR-ADMIN-PROD-04: 상품 활성화/비활성화**
- **입력**:
  - 상품 ID
  - 상태 (ACTIVE/INACTIVE)
- **처리 과정**:
  1. 상품 존재 여부 확인
  2. 상품 상태 업데이트
- **출력**: 업데이트된 상품 상태
- **효과**:
  - ACTIVE: 사용자에게 노출
  - INACTIVE: 사용자에게 비노출, 장바구니/주문 불가
- **관련 유스케이스**: [UC-ADMIN-PROD-04](user-stories.md#uc-admin-prod-04-상품-활성화)

#### 비기능 요구사항

**NFR-ADMIN-PROD-PERF: 성능**
- 상품 등록/수정: 평균 300ms 이내 응답
- 상품 조회: 평균 200ms 이내 응답

**NFR-ADMIN-PROD-RELI: 안정성**
- **데이터 유효성**: 필수 필드 검증 및 데이터 정합성 보장
- **가격 정확성**: 가격은 양수, 소수점 2자리까지 지원
- **참조 무결성**: 카테고리 삭제 시 해당 상품 처리 정책 필요

### FR-ADMIN-CAT: 상품 카테고리 관리

| ID | 요구사항 | 우선순위 | 상태 |
|----|---------|---------|------|
| **FR-ADMIN-CAT-01** | 시스템은 카테고리를 생성할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-CAT-02** | 시스템은 카테고리를 수정할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-CAT-03** | 시스템은 카테고리를 삭제할 수 있어야 함 | MUST | ⏳ 대기 |

#### 상세 설명

**FR-ADMIN-CAT-01: 카테고리 생성**
- **입력**:
  - 카테고리명
  - 설명 (선택)
  - 정렬 순서
- **처리 과정**:
  1. 입력 데이터 유효성 검증
  2. 중복 카테고리명 확인
  3. 카테고리 생성
- **출력**: 생성된 카테고리 ID 및 정보
- **참고**: 카테고리는 단일 레벨 구조 (계층 없음)
- **관련 유스케이스**: [UC-ADMIN-CAT-01](user-stories.md#uc-admin-cat-01-카테고리-생성)

**FR-ADMIN-CAT-02: 카테고리 수정**
- **입력**:
  - 카테고리 ID
  - 수정할 필드 (카테고리명, 설명, 정렬 순서)
- **처리 과정**:
  1. 카테고리 존재 여부 확인
  2. 입력 데이터 유효성 검증
  3. 카테고리 정보 업데이트
- **출력**: 수정된 카테고리 정보
- **관련 유스케이스**: [UC-ADMIN-CAT-02](user-stories.md#uc-admin-cat-02-카테고리-수정)

**FR-ADMIN-CAT-03: 카테고리 삭제**
- **입력**: 카테고리 ID
- **처리 과정**:
  1. 카테고리 존재 여부 확인
  2. 연결된 상품 존재 여부 확인
  3. 삭제 또는 에러 반환
- **출력**: 삭제 확인 메시지
- **제약사항**: 상품이 연결된 경우 삭제 불가 (또는 상품을 다른 카테고리로 이동 후 삭제)
- **관련 유스케이스**: [UC-ADMIN-CAT-03](user-stories.md#uc-admin-cat-03-카테고리-삭제)

#### 비기능 요구사항

**NFR-ADMIN-CAT-PERF: 성능**
- 카테고리 생성/수정/삭제: 평균 200ms 이내 응답
- 카테고리 조회: 평균 150ms 이내 응답

**NFR-ADMIN-CAT-RELI: 안정성**
- **참조 무결성**: 카테고리 삭제 시 연관 상품 처리
- **중복 방지**: 카테고리명 중복 불가

### FR-ADMIN-OPT: 상품 옵션 관리

| ID | 요구사항 | 우선순위 | 상태 |
|----|---------|---------|------|
| **FR-ADMIN-OPT-01** | 시스템은 옵션 그룹을 생성할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-OPT-02** | 시스템은 옵션 그룹을 수정/삭제할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-OPT-03** | 시스템은 옵션값을 생성할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-OPT-04** | 시스템은 옵션값을 수정/삭제할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-OPT-05** | 시스템은 상품에 옵션을 연결할 수 있어야 함 | MUST | ⏳ 대기 |

#### 상세 설명

**FR-ADMIN-OPT-01: 옵션 그룹 생성**
- **입력**:
  - 옵션 그룹명 (예: "색상", "사이즈")
  - 설명 (선택)
- **처리 과정**:
  1. 입력 데이터 유효성 검증
  2. 중복 옵션 그룹명 확인
  3. 옵션 그룹 생성
- **출력**: 생성된 옵션 그룹 ID 및 정보
- **예시**: 색상, 사이즈, 용량, 재질 등
- **관련 유스케이스**: [UC-ADMIN-OPT-01](user-stories.md#uc-admin-opt-01-옵션-그룹-생성)

**FR-ADMIN-OPT-02: 옵션 그룹 수정/삭제**
- **수정 입력**:
  - 옵션 그룹 ID
  - 수정할 필드 (그룹명, 설명)
- **삭제 입력**: 옵션 그룹 ID
- **처리 과정**:
  1. 옵션 그룹 존재 여부 확인
  2. 연결된 옵션값 확인 (삭제 시)
  3. 업데이트 또는 삭제
- **출력**: 수정된 정보 또는 삭제 확인
- **제약사항**: 옵션값이 연결된 그룹은 삭제 불가
- **관련 유스케이스**: [UC-ADMIN-OPT-02](user-stories.md#uc-admin-opt-02-옵션-그룹-수정)

**FR-ADMIN-OPT-03: 옵션값 생성**
- **입력**:
  - 옵션 그룹 ID
  - 옵션값 (예: "빨강", "파랑", "S", "M", "L")
  - 추가 가격 (기본값: 0)
- **처리 과정**:
  1. 옵션 그룹 존재 여부 확인
  2. 입력 데이터 유효성 검증
  3. 중복 옵션값 확인 (동일 그룹 내)
  4. 옵션값 생성
- **출력**: 생성된 옵션값 ID 및 정보
- **관련 유스케이스**: [UC-ADMIN-OPT-03](user-stories.md#uc-admin-opt-03-옵션값-생성)

**FR-ADMIN-OPT-04: 옵션값 수정/삭제**
- **수정 입력**:
  - 옵션값 ID
  - 수정할 필드 (옵션값, 추가 가격)
- **삭제 입력**: 옵션값 ID
- **처리 과정**:
  1. 옵션값 존재 여부 확인
  2. 주문 내역 확인 (삭제 시)
  3. 업데이트 또는 삭제
- **출력**: 수정된 정보 또는 삭제 확인
- **제약사항**: 주문 내역이 있는 옵션값은 소프트 삭제 권장
- **관련 유스케이스**: [UC-ADMIN-OPT-04](user-stories.md#uc-admin-opt-04-옵션값-수정)

**FR-ADMIN-OPT-05: 상품-옵션 연결**
- **입력**:
  - 상품 ID
  - 옵션 그룹 ID 목록
- **처리 과정**:
  1. 상품 존재 여부 확인
  2. 옵션 그룹 존재 여부 확인
  3. 상품에 옵션 그룹 연결
  4. 옵션 조합별 초기 재고 0 설정
- **출력**: 연결된 옵션 정보
- **예시**: 티셔츠 상품에 "색상" + "사이즈" 옵션 그룹 연결
- **관련 유스케이스**: [UC-ADMIN-OPT-05](user-stories.md#uc-admin-opt-05-상품-옵션-연결)

#### 비기능 요구사항

**NFR-ADMIN-OPT-PERF: 성능**
- 옵션 생성/수정/삭제: 평균 200ms 이내 응답
- 옵션 조회: 평균 150ms 이내 응답

**NFR-ADMIN-OPT-RELI: 안정성**
- **데이터 정합성**: 상품-옵션 관계 무결성 보장
- **가격 정확성**: 추가 가격은 0 이상
- **중복 방지**: 동일 그룹 내 옵션값 중복 불가

### FR-ADMIN-INV: 재고 관리

| ID | 요구사항 | 우선순위 | 상태 |
|----|---------|---------|------|
| **FR-ADMIN-INV-01** | 시스템은 재고 수량을 조회할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-INV-02** | 시스템은 재고를 입고 처리할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-INV-03** | 시스템은 재고를 조정할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-INV-04** | 시스템은 재고 이력을 조회할 수 있어야 함 | SHOULD | ⏳ 대기 |

#### 상세 설명

**FR-ADMIN-INV-01: 재고 조회**
- **입력**:
  - 상품 ID (선택)
  - 옵션 ID (선택)
  - 재고 상태 필터 (품절/정상)
- **처리 과정**:
  1. 필터 조건에 맞는 재고 조회
  2. 상품/옵션 정보와 함께 반환
- **출력**: 재고 목록 (상품명, 옵션, 현재 수량, 확보 수량, 가용 수량)
- **관련 유스케이스**: [UC-ADMIN-INV-01](user-stories.md#uc-admin-inv-01-재고-조회)

**FR-ADMIN-INV-02: 재고 입고**
- **입력**:
  - 상품 ID
  - 옵션 ID (선택)
  - 입고 수량
  - 입고 사유/메모
- **처리 과정**:
  1. 상품/옵션 존재 여부 확인
  2. 입고 수량 유효성 검증 (양수)
  3. 재고 수량 증가
  4. 재고 이력 기록
- **출력**: 업데이트된 재고 정보
- **관련 유스케이스**: [UC-ADMIN-INV-02](user-stories.md#uc-admin-inv-02-재고-입고)

**FR-ADMIN-INV-03: 재고 조정**
- **입력**:
  - 상품 ID
  - 옵션 ID (선택)
  - 조정 수량 (양수/음수)
  - 조정 사유
- **처리 과정**:
  1. 상품/옵션 존재 여부 확인
  2. 조정 후 재고가 0 미만인지 확인
  3. 재고 수량 조정
  4. 재고 이력 기록
- **출력**: 조정된 재고 정보
- **제약사항**: 조정 후 재고는 0 이상이어야 함
- **관련 유스케이스**: [UC-ADMIN-INV-03](user-stories.md#uc-admin-inv-03-재고-조정)

**FR-ADMIN-INV-04: 재고 이력 조회**
- **입력**:
  - 상품 ID (선택)
  - 옵션 ID (선택)
  - 기간 (시작일, 종료일)
  - 이력 유형 (입고/출고/조정/확보/복원)
- **처리 과정**:
  1. 필터 조건에 맞는 이력 조회
  2. 페이지네이션 처리
- **출력**: 재고 이력 목록 (일시, 유형, 변경 수량, 이전 수량, 이후 수량, 사유)
- **활용**: 재고 추적, 재고 분석, 감사
- **관련 유스케이스**: [UC-ADMIN-INV-04](user-stories.md#uc-admin-inv-04-재고-이력-조회)

#### 비기능 요구사항

**NFR-ADMIN-INV-PERF: 성능**
- 재고 조회: 평균 200ms 이내 응답
- 재고 입고/조정: 평균 300ms 이내 응답
- 재고 이력 조회: 평균 500ms 이내 응답 (페이지네이션)

**NFR-ADMIN-INV-RELI: 안정성**
- **재고 정확성**: 재고 수량 정확성 보장 (동시성 제어)
- **이력 추적**: 모든 재고 변동 이력 기록 필수
- **음수 방지**: 재고는 항상 0 이상 유지

### FR-ADMIN-COUPON: 쿠폰 발급 관리

| ID | 요구사항 | 우선순위 | 상태 |
|----|---------|---------|------|
| **FR-ADMIN-COUPON-01** | 시스템은 쿠폰을 생성할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-COUPON-02** | 시스템은 쿠폰을 수정할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-COUPON-03** | 시스템은 쿠폰을 삭제/비활성화할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-COUPON-04** | 시스템은 쿠폰 발급 현황을 조회할 수 있어야 함 | SHOULD | ⏳ 대기 |

#### 상세 설명

**FR-ADMIN-COUPON-01: 쿠폰 생성**
- **입력**:
  - 쿠폰명
  - 쿠폰 코드 (자동 생성 또는 수동 입력)
  - 할인 유형 (퍼센트/금액)
  - 할인 값 (예: 10% 또는 5000원)
  - 발급 수량
  - 유효기간 (시작일, 종료일)
  - 최소 주문 금액 (선택)
- **처리 과정**:
  1. 입력 데이터 유효성 검증
  2. 쿠폰 코드 중복 확인
  3. 쿠폰 생성
  4. 초기 상태: ACTIVE
- **출력**: 생성된 쿠폰 ID 및 정보
- **관련 유스케이스**: [UC-ADMIN-COUPON-01](user-stories.md#uc-admin-coupon-01-쿠폰-생성)

**FR-ADMIN-COUPON-02: 쿠폰 수정**
- **입력**:
  - 쿠폰 ID
  - 수정할 필드 (쿠폰명, 할인 값, 발급 수량, 유효기간, 최소 주문 금액)
- **처리 과정**:
  1. 쿠폰 존재 여부 확인
  2. 입력 데이터 유효성 검증
  3. 발급된 쿠폰 수보다 발급 수량을 줄일 수 없음 확인
  4. 쿠폰 정보 업데이트
- **출력**: 수정된 쿠폰 정보
- **제약사항**:
  - 쿠폰 코드는 수정 불가
  - 발급 수량은 이미 발급된 수량보다 작게 설정 불가
- **관련 유스케이스**: [UC-ADMIN-COUPON-02](user-stories.md#uc-admin-coupon-02-쿠폰-수정)

**FR-ADMIN-COUPON-03: 쿠폰 삭제/비활성화**
- **입력**:
  - 쿠폰 ID
  - 상태 (INACTIVE) 또는 삭제
- **처리 과정**:
  1. 쿠폰 존재 여부 확인
  2. 발급 이력 확인
  3. 비활성화 또는 소프트 삭제
- **출력**: 상태 변경 확인
- **효과**:
  - INACTIVE: 신규 발급 중단, 기존 발급된 쿠폰은 사용 가능
  - 삭제: 발급 이력이 없는 경우에만 물리적 삭제 가능
- **관련 유스케이스**: [UC-ADMIN-COUPON-03](user-stories.md#uc-admin-coupon-03-쿠폰-비활성화)

**FR-ADMIN-COUPON-04: 쿠폰 발급 현황 조회**
- **입력**:
  - 쿠폰 ID (선택)
  - 기간 (선택)
- **처리 과정**:
  1. 쿠폰 목록 조회
  2. 각 쿠폰별 발급 통계 집계
- **출력**:
  - 쿠폰 정보
  - 총 발급 수량
  - 발급된 수량
  - 잔여 수량
  - 사용된 수량
  - 사용률
- **관련 유스케이스**: [UC-ADMIN-COUPON-04](user-stories.md#uc-admin-coupon-04-발급-현황-조회)

#### 비기능 요구사항

**NFR-ADMIN-COUPON-PERF: 성능**
- 쿠폰 생성/수정: 평균 200ms 이내 응답
- 쿠폰 발급 현황 조회: 평균 300ms 이내 응답

**NFR-ADMIN-COUPON-RELI: 안정성**
- **코드 유일성**: 쿠폰 코드는 시스템 전체에서 유일해야 함
- **수량 정합성**: 발급 수량과 실제 발급 이력의 일관성 보장
- **데이터 무결성**: 이미 발급된 쿠폰에 영향을 주지 않는 수정만 허용

### FR-ADMIN-ORDER: 주문 관리

| ID | 요구사항 | 우선순위 | 상태 |
|----|---------|---------|------|
| **FR-ADMIN-ORDER-01** | 시스템은 전체 주문 목록을 조회할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-ORDER-02** | 시스템은 주문 상세를 조회할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-ORDER-03** | 시스템은 주문 상태를 변경할 수 있어야 함 | MUST | ⏳ 대기 |
| **FR-ADMIN-ORDER-04** | 시스템은 주문을 취소할 수 있어야 함 | MUST | ⏳ 대기 |

#### 상세 설명

**FR-ADMIN-ORDER-01: 전체 주문 목록 조회**
- **입력**:
  - 페이지네이션 (page, limit)
  - 필터 (주문 상태, 기간, 사용자 ID, 상품 ID)
  - 정렬 (주문일시, 결제 금액)
- **처리 과정**:
  1. 필터 조건에 맞는 주문 조회
  2. 정렬 및 페이지네이션 적용
- **출력**: 주문 목록 (주문 ID, 사용자, 주문일시, 결제 금액, 주문 상태)
- **관련 유스케이스**: [UC-ADMIN-ORDER-01](user-stories.md#uc-admin-order-01-주문-목록-조회)

**FR-ADMIN-ORDER-02: 주문 상세 조회**
- **입력**: 주문 ID
- **처리 과정**:
  1. 주문 정보 조회
  2. 주문 상품 목록 조회
  3. 결제 정보 조회
  4. 쿠폰 사용 정보 조회
  5. 사용자 정보 조회
- **출력**:
  - 주문 전체 정보
  - 주문 상품 목록 (상품명, 옵션, 수량, 단가, 소계)
  - 결제 정보 (결제 금액, 할인 금액, 최종 금액)
  - 쿠폰 정보
  - 주문 상태 이력
- **관련 유스케이스**: [UC-ADMIN-ORDER-02](user-stories.md#uc-admin-order-02-주문-상세-조회)

**FR-ADMIN-ORDER-03: 주문 상태 변경**
- **입력**:
  - 주문 ID
  - 변경할 상태
  - 사유 (선택)
- **처리 과정**:
  1. 주문 존재 여부 확인
  2. 현재 상태 확인
  3. 상태 전이 규칙 검증 ([requirements.md - FR-ORDER-04](requirements.md#fr-order-04-주문-상태-관리) 참조)
  4. 주문 상태 업데이트
  5. 상태 변경 이력 기록
- **출력**: 변경된 주문 상태
- **제약사항**: 허용된 상태 전이만 가능
- **관련 유스케이스**: [UC-ADMIN-ORDER-03](user-stories.md#uc-admin-order-03-주문-상태-변경)

**FR-ADMIN-ORDER-04: 주문 취소**
- **입력**:
  - 주문 ID
  - 취소 사유
- **처리 과정**:
  1. 주문 상태 확인 (PENDING 또는 COMPLETED만 취소 가능)
  2. 결제 완료 상태인 경우 환불 처리 (가정)
  3. 재고 복원 ([requirements.md - FR-PAYMENT-03](requirements.md#fr-payment-03-재고-복원) 참조)
  4. 쿠폰 복원 (사용된 쿠폰을 AVAILABLE 상태로 변경)
  5. 주문 상태를 CANCELLED로 변경
  6. 취소 이력 기록
- **출력**: 취소 확인 및 환불 정보
- **관련 유스케이스**: [UC-ADMIN-ORDER-04](user-stories.md#uc-admin-order-04-주문-취소)

#### 비기능 요구사항

**NFR-ADMIN-ORDER-PERF: 성능**
- 주문 목록 조회: 평균 300ms 이내 응답 (페이지네이션)
- 주문 상세 조회: 평균 200ms 이내 응답
- 주문 상태 변경: 평균 300ms 이내 응답

**NFR-ADMIN-ORDER-RELI: 안정성**
- **상태 정합성**: 주문 상태 전이 규칙 엄격히 준수
- **재고 정합성**: 주문 취소 시 재고 복원 보장
- **쿠폰 정합성**: 주문 취소 시 쿠폰 복원 보장
- **이력 추적**: 모든 상태 변경 이력 기록
